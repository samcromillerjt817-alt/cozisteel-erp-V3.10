# ADR-001 — Princípios Arquiteturais do Cozisteel ERP

- **Status**: Aceito
- **Data**: 2026-07-09
- **Contexto do projeto**: definido durante o planejamento do roadmap de evolução V4 (ver
  `docs/adr/` para decisões futuras e o histórico de decisões ao final deste documento)

## Contexto

O Cozisteel ERP está evoluindo de um sistema com lógica de negócio espalhada em rotas de API para uma
arquitetura industrial moderna — camada de Services, eventos de domínio, máquina de estados, MRP, reserva
de estoque, rastreabilidade por lote e, eventualmente, um módulo Financeiro transversal. Esse processo
acontece em fases (ver roadmap de 12 fases), ao longo de múltiplas sessões de desenvolvimento.

Este documento existe para que nenhuma decisão de uma fase específica traia a arquitetura final do
sistema. Ele é a referência que qualquer decisão técnica futura — em qualquer fase, em qualquer módulo —
deve respeitar. Quando um princípio aqui e a conveniência de curto prazo entrarem em conflito, o princípio
vence.

## Princípios

### 1. Toda lógica de negócio pertence aos Services
Rotas (`src/app/api/**/route.ts`) fazem apenas: validar entrada, verificar autenticação/permissão, chamar
o Service responsável, devolver a resposta. Nenhuma regra de negócio vive numa rota.

### 2. Módulos se comunicam por Eventos de Domínio
Nenhum módulo altera diretamente o dado de outro módulo. Toda integração passa por Services, Eventos de
Domínio, ou referência de documento (`originDocumentType`/`originDocumentId`) — nunca uma chamada direta
que acopla um módulo à implementação interna de outro.

### 3. Operações críticas são transacionais
Qualquer operação que grave em mais de uma tabela como parte de uma única ação de negócio (ex: concluir
uma OP: baixar matéria-prima + criar movimentação + dar entrada de produto acabado + atualizar a OP) usa
`db.$transaction`. Nunca deixar o sistema num estado intermediário se uma etapa falhar.

### 4. Documento Único, Responsabilidade Única
Cada documento do ERP tem exatamente uma responsabilidade de negócio — nunca acumula funções de
documentos diferentes:

| Documento | Responsabilidade |
|---|---|
| Orçamento | Negociar |
| Pedido de Venda | Formalizar a venda |
| Ordem de Produção | Controlar a fabricação |
| Requisição | Registrar uma necessidade interna |
| Pedido de Compra | Formalizar a compra |
| Recebimento | Registrar a entrada física |
| Movimentação de Estoque | Controlar movimentações |
| Contas a Pagar/Receber | Controlar obrigações financeiras |

### 5. Fonte Única da Verdade (Single Source of Truth)
Nenhuma informação crítica existe duplicada — cada módulo é dono dos seus dados, os demais consomem, nunca
replicam:

- **Estoque** controla saldo.
- **Engenharia** controla a BOM (exceto o snapshot congelado dentro de uma Ordem de Produção já criada).
- **Máquina de Estados** controla status.
- **Auditoria** controla histórico.
- **Financeiro** controla títulos financeiros.

### 6. Rastreabilidade Total
Todo documento tem origem e destino identificáveis — é sempre possível navegar para frente e para trás
na cadeia completa:

```
Cliente → Orçamento → Pedido de Venda → Ordem de Produção → Requisição → Pedido de Compra →
Recebimento → Estoque → Faturamento → Financeiro
```

Nenhum documento fica órfão (sem origem rastreável ou sem destino conhecido).

### 7. Toda alteração de status passa pela Máquina de Estados
Nenhuma entidade com ciclo de vida (status) valida transição com um `if`/array solto próprio — todas usam
a infraestrutura compartilhada de workflow (`checkTransition()` / `WorkflowService`, conforme a fase do
roadmap). Toda transição gera um registro em `StatusHistory` — **cumprido integralmente desde a Fase 2.1**
(2026-07-09): os 6 domínios com ciclo de vida (Orçamento, Pedido de Venda, Ordem de Produção, Requisição,
Pedido de Compra, Revisão de BOM) gravam `StatusHistory` logo após `checkTransition()` validar a transição.

### 8. Toda movimentação é auditável
Toda operação crítica registra usuário, IP, navegador, data, valores anteriores e novos — não apenas um
texto livre de descrição. **Cumprido parcialmente**: `AuditLog.beforeValue`/`afterValue` existem desde a
Fase 2.1 (2026-07-09) e são preenchidos nas 4 transições de status que já auditavam (Orçamento, Pedido de
Venda, Requisição, Pedido de Compra). Fora dos pontos de transição de status, a cobertura de auditoria
segue com as mesmas lacunas já catalogadas como débito técnico na Revisão de Fim de Fase da Fase 1
(alguns creates/updates sem log de auditoria nenhum) — este princípio não foi levado a 100% de cobertura
nesta rodada, só ao ponto que a Fase 5.9 se propôs a resolver.

### 9. Parametrização antes de Código
Sempre que uma regra puder variar entre instalações/empresas, ela é configurável via `SystemSetting`, não
fixa em código com condicional. Exemplos: aprovação obrigatória de compra, permitir estoque negativo,
fluxos de aprovação, numeração de documentos, Centros de Custo, workflows. ~~Geração automática de OP~~
removido desta lista em 2026-07-09 — decisão oficial: a geração automática de Ordem de Produção na
aprovação do Orçamento/Pedido de Venda é o fluxo definitivo do Cozisteel, não um candidato a toggle de
`SystemSetting` (ver "Auditoria de Débito Estrutural Pendente" para o histórico da decisão).

### 10. Financeiro é transversal
O módulo Financeiro nunca inicia processo por conta própria — é alimentado pelos demais módulos
(Comercial, Compras, Produção, Estoque) via Services, Eventos de Domínio e referências de documento.

### 11. Artefatos derivados de documentação/visualização não podem residir dentro do corpus analisado pelo Graphify
Nenhuma saída gerada a partir do grafo (exports para Obsidian/Markdown, visualizações, relatórios
derivados) pode ser gravada dentro do diretório varrido por `graphify update .`. `graphify update .`
reingere qualquer arquivo dentro do projeto como fonte — se a própria saída do grafo for gravada ali,
a rodada seguinte reprocessa essa saída como se fosse código, formando um loop de realimentação que
infla o grafo indefinidamente (visto na prática: 1313 → 9584 nós em 3 execuções, ver log de decisões).
Regra prática: exports de documentação/visualização sempre gravam fora da raiz do projeto (ex:
`~/Cozisteel-Knowledge`); `.graphifyignore` na raiz do projeto exclui explicitamente qualquer artefato
derivado que ainda assim apareça dentro do repo (`graphify-out/`, `graphify-obsidian/`, `.obsidian/`).

### 12. O Graphify é fonte de engenharia, não documentação complementar (2026-07-10)
O grafo do projeto (`graphify-out/`) deixa de ser um artefato acessório e passa a ser uma das
principais fontes de consulta para continuidade, integridade arquitetural e recuperação de contexto
durante todo o desenvolvimento. Disciplina obrigatória a partir desta data:

1. **Antes** de iniciar qualquer nova fase, subetapa, ADR, implementação, refatoração ou alteração
   estrutural: consultar o Graphify para localizar ADRs relacionados, decisões arquiteturais
   anteriores, entidades envolvidas, relacionamentos, Services, Repositories, fluxos, dependências,
   regras de negócio e impactos cruzados.
2. Sempre que houver dúvida sobre continuidade, reuso de código, integridade arquitetural ou impacto de
   uma alteração: consultar o Graphify antes de qualquer decisão técnica.
3. **Durante** a implementação: usar o Graphify continuamente como mapa de navegação e recuperação de
   contexto do ERP.
4. **Ao término** de cada subetapa implementada: `graphify update .` é obrigatório, refletindo novas
   entidades, relacionamentos, Services, Repositories, fluxos, regras de negócio, ADRs e alterações
   arquiteturais aprovadas — não apenas ao fim de uma fase inteira.
5. Objetivo: manter o Graphify permanentemente sincronizado com a evolução do projeto, para que
   qualquer continuidade futura preserve o contexto arquitetural na íntegra.

**Ordem de consulta oficial do projeto**: Código implementado → ADRs aprovados → Graphify. Os três
permanecem permanentemente sincronizados; qualquer divergência encontrada entre eles é corrigida
imediatamente (documentação e grafo atualizados para refletir fielmente a implementação aprovada), não
deixada para depois.

**Diagnóstico técnico registrado (2026-07-10)** — o usuário pediu uma investigação formal antes de
continuar tratando o Graphify como mapa arquitetural principal, especificamente sobre os ADRs:

| Pergunta | Resposta |
|---|---|
| Os ADRs estão realmente sendo indexados? | **Não.** `graphify update .` só faz extração de AST de código-fonte. Confirmado por consulta direta ao grafo: nenhum nó em `graphify-out/graph.json` tem `source` apontando para um arquivo `.md` de `docs/adr/`, e conceitos exclusivos de um ADR (ex. "Recharts", "Dashboard modular por widgets", só existentes no ADR-017) não aparecem no grafo em nenhuma forma. |
| Apenas o hash falha? | **Não, é mais profundo que isso.** O warning `file_hash requires a file, got: ...` (case-sensitivity de path neste ambiente WSL/Windows, confirmado reproduzindo o erro diretamente) é um sintoma cosmético adicional, não a causa raiz. Mesmo sem esse warning, o conteúdo de markdown nunca seria absorvido, porque a extração semântica de documentos exige o pipeline com LLM (`GEMINI_API_KEY`/`GOOGLE_API_KEY`), não configurado neste projeto — só a extração de AST de código roda hoje. |
| O conteúdo continua sendo absorvido? | **Não.** Os únicos nós "ADR-00XX" que existem no grafo são citações rasas (`docref`, grau 1) extraídas de comentários de código-fonte que mencionam o número do ADR (ex. um comentário `// ADR-017` em `route.ts` gera um nó apontando de volta para esse arquivo de código — nunca para o conteúdo do ADR). |
| Existe perda de informação? | **Sim, mas não é uma regressão recente** — é uma lacuna estrutural que sempre existiu neste projeto, só nunca formalmente diagnosticada. Ao longo de toda esta iniciativa, sempre que o conteúdo de um ADR foi necessário, a leitura direta do arquivo (`Read`) foi usada — nunca `graphify query` para o conteúdo do ADR em si. Essa prática já estava correta; agora está formalmente justificada. |
| O problema é causado pelo ambiente WSL/Windows? | **Parcialmente.** O warning específico de `file_hash` (case-sensitivity) é causado pelo ambiente — confirmado que o filesystem aqui é case-sensitive e o path é normalizado para minúsculas internamente antes da checagem de hash. Mas a ausência de indexação SEMÂNTICA de markdown é uma limitação de configuração (chave de LLM ausente) que existiria em qualquer ambiente, não é causada pelo WSL/Windows. |

**Implicação prática, daqui em diante**: o Graphify (Princípio 12 acima) continua sendo consultado
antes de qualquer fase/subetapa/ADR/implementação — mas para localizar QUAL código já cita um ADR ou
entidade (via `graphify explain`/`query`/`path` sobre código), nunca como substituto da leitura direta
do conteúdo de um ADR. Ler o ADR relevante via `Read` continua obrigatório sempre que a decisão
depender do conteúdo doutrinário do documento, não só da sua existência.

## Consequências

- Toda fase do roadmap de evolução é avaliada contra estes 12 princípios antes de ser considerada
  concluída, não só contra o próprio escopo da fase (ver diretriz de arquitetura de longo prazo).
- Decisões que violem um destes princípios por conveniência de prazo devem ser sinalizadas explicitamente
  como exceção temporária, nunca implementadas silenciosamente.
- Este documento é vivo — toda decisão arquitetural relevante nova é registrada no log abaixo, e o
  grafo do projeto (graphify) é atualizado junto (Princípio 12).

## Revisão de Fim de Fase — Fase 1 (2026-07-09)

Fase 1 concluída: toda a lógica de negócio foi extraída das ~40 rotas de API para Services, com uma
camada de Repository (Prisma, por enquanto) entre Service e banco. Revisão objetiva contra os critérios
combinados antes de iniciar a Fase 1:

- **Nenhuma lógica de negócio importante permanece nas rotas.** Confirmado por varredura: nenhuma rota em
  `src/app/api/**/route.ts` chama `db.<model>.(findMany|findUnique|findFirst|create|update|delete|
  deleteMany|updateMany|count|upsert|aggregate|groupBy)` diretamente. Toda rota apenas autentica, valida
  entrada (Zod, quando existente) e delega ao Service correspondente. As únicas rotas que não têm
  Service+Repository próprios são as de geração de PDF (`*/pdf/route.ts`, `*/transport-pdf/route.ts`) —
  já delegavam 100% a `PdfService`, que é anterior a esta fase e não precisou mudar.
- **Cada Service tem responsabilidade única.** 22 Services, um por agregado/caso de uso (Categoria,
  Cliente, Material, Fornecedor, Produto, Usuário, Sequência, Sistema, Patch, Storage, Setting, Report,
  Audit, Numbering, Quote, SalesOrder, ProductionOrder, Requisition, PurchaseOrder, Stock, Dashboard,
  PDF). Nenhum "Service genérico" acumulando responsabilidades de mais de um domínio.
- **Sem lógica duplicada.** Os 6 padrões duplicados identificados no levantamento inicial (checagem de
  erro por `catch`, existence-check-then-404, checagem de unicidade, strip de campos relacionais antes
  de update, upsert de tabela de junção, flag booleana exclusiva, cálculo de volume) foram centralizados
  em `handleRouteError()`, `BaseRepository`, e helpers privados dentro do Service dono de cada regra.
  `getTodayDate()` (duplicado antes da Fase 1) já estava consolidado em `formatDate()` (`src/lib/format.ts`).
- **Sem acoplamento desnecessário entre módulos.** Nenhum Service importa o Repository de outro módulo
  para *escrever* dado que não é seu — os únicos acessos cruzados são leituras de validação (ex:
  `QuoteService` lê `clientRepository` só pra popular snapshot; `RequisitionService` lê
  `productionOrderRepository`/`materialRepository`/`supplierRepository` só pra validar existência) ou
  escrita através do Repository que já é dono legítimo do campo (`StockService` grava `stockQty` via
  `materialRepository`/`productRepository`, que são os donos estruturais dessa coluna no schema — a
  auditoria de estoque em si, `StockMovement`, é exclusiva do `StockRepository`). As três integrações de
  negócio entre agregados diferentes (Orçamento→Ordem de Produção, Orçamento→Pedido de Venda,
  Requisição→Pedido de Compra) são sempre chamada Service-a-Service (`quoteService` chama
  `productionOrderService`/`salesOrderService`; `requisitionService` chama `purchaseOrderService`),
  nunca uma Repository cruzando a fronteira do agregado.
- **Estrutura pronta para Máquina de Estados (Fase 2).** Toda transição de status hoje passa por um
  método único de Service (`changeStatus`/`update`), nunca um `if`/array solto na rota. Os módulos que já
  tinham restrição de transição antes da Fase 1 (Requisição, Pedido de Compra) usam o helper compartilhado
  `checkTransition()` (criado ainda antes da Fase 1, preservado). Os módulos sem restrição hoje (Orçamento,
  Pedido de Venda) foram mantidos exatamente como estavam — sem introduzir `checkTransition` ou novas
  regras de transição, porque isso seria uma mudança de comportamento fora do escopo "estrutural" da
  Fase 1. Generalizar isso é o trabalho da Fase 2, e o ponto de entrada (`changeStatus`) já está isolado
  e pronto para receber essa validação sem precisar tocar a rota de novo.
- **Estrutura pronta para Eventos de Domínio (Fase 3).** As duas operações transacionais mais críticas do
  sistema (conclusão de Ordem de Produção com baixa de matéria-prima/entrada de produto acabado;
  recebimento de Pedido de Compra com entrada de estoque) já rodam dentro de `db.$transaction`, isoladas
  num único método de Repository (`completeAndConsumeStock`, `receiveItems`). As três chamadas
  Service-a-Service entre agregados diferentes (ver item de acoplamento acima) são o ponto exato que vira
  publicação de evento na Fase 3 — a troca é local a esses três métodos, sem precisar reabrir nenhuma rota.
- **Estrutura pronta para Reserva de Estoque/MRP/Financeiro.** Nenhuma decisão desta fase acopla a lógica
  de negócio a um formato de armazenamento que dificulte isso: `StockService`/`StockRepository` isolam
  toda leitura/escrita de saldo num único lugar (hoje ainda apoiado nas colunas `stockQty` de
  `Material`/`Product`, mas por trás de uma interface de Service que pode evoluir sem mexer nas rotas);
  `RequisitionService`/`PurchaseOrderService` já carregam o vocabulário de Centro de Custo/rastreabilidade
  implícito na cadeia Requisição→Pedido de Compra→Recebimento sem hardcode que impeça extensão futura.
- **Todos os princípios do ADR-001 seguidos.** Nenhuma exceção temporária foi necessária nesta fase.
- **Débito técnico catalogado** (não é bloqueador, decisão consciente de não misturar com a extração
  estrutural, conforme já combinado antes da Fase 1): validação Zod ausente em quase todas as rotas PUT;
  alguns creates/updates (Produto, Categoria) sem log de auditoria enquanto Material/Fornecedor logam;
  checagem de permissão inconsistente entre rotas irmãs (ex: vínculo produto↔matéria-prima só pede login,
  o produto em si pede permissão de módulo); soft-delete em Produto vs. hard-delete-com-guarda em
  Material/Fornecedor; `interface Quote` do frontend (`src/app/page.tsx:37`) sem o campo `salesOrder`
  (gera erro de `tsc` pré-existente, confirmado não introduzido por esta fase).
- **Achado operacional, resolvido antes da Fase 2** (ver princípio 11 e log de decisões abaixo): loop de
  realimentação entre `graphify update .` e o watcher `watch-graphify.sh`/`graphify_to_obsidian.py` do
  usuário, corrigido na raiz (exportador movido para fora do repo + `.graphifyignore`), não apenas
  contornado a cada rodada.

**Conclusão**: Fase 1 atende a todos os critérios combinados. Nenhum ponto identificado compromete a
Fase 2 (Máquina de Estados) ou a Fase 3 (Eventos de Domínio) — não foi necessário redesenhar nada
durante a implementação.

## Auditoria de Débito Estrutural Pendente (2026-07-09)

Antes de validar a Fase 6 (MRP), o usuário pediu uma revisão de todo o roadmap (Fases 1-12) procurando
infraestrutura estrutural prevista originalmente mas nunca implementada — risco real de retrabalho nas
fases seguintes. Varredura feita contra o schema real e o código real (não contra a memória do que foi
planejado). Achados:

1. **Model `CostCenter` nunca criado** — **resolvido em 2026-07-09**, ver
   [ADR-008](./ADR-008-infraestrutura-financeira.md): registrado como infraestrutura oportunista,
   condicionada à existência de um domínio real de Centro de Custos **e** a uma fase que já vá alterar a
   entidade em questão (Fase 7 para `Requisition`, Fase 8 para `PurchaseOrder`/`ProductionOrder`). Nenhuma
   implementação preventiva.

2. **Tabela `StatusHistory` nunca criada** (prevista na Fase 2) — **implementada em 2026-07-09 (Fase
   2.1), Opção A aprovada e executada.** Model `StatusHistory` adicionado ao schema (mesmo padrão
   append-only com discriminador já usado em `AuditLog`/`StockMovement`); `statusHistoryService.record()`
   (novo, write-only, sem consulta própria por instrução explícita) chamado logo após `checkTransition()`
   validar a transição nos 6 domínios com ciclo de vida (Orçamento, Pedido de Venda, Ordem de Produção,
   Requisição, Pedido de Compra, Revisão de BOM). Nenhuma regra de negócio nova, nenhuma mudança na
   Máquina de Estados, nenhuma mudança de comportamento visível — confirmado pelos 43 testes automatizados
   existentes, todos passando sem alteração. Princípio 7 corrigido para refletir cumprimento integral.

3. **Auditoria "antes/depois" (`beforeValue`/`afterValue`) nunca implementada** (prevista na Fase 3,
   Princípio 8) — **implementada em 2026-07-09 (junto da Fase 2.1), no escopo que o usuário definiu:
   só os mesmos pontos de transição de status já tocados para o `StatusHistory`.** `AuditLog` ganhou
   `beforeValue`/`afterValue` (`Json?`, opcionais, compatíveis com todo registro antigo). Populados nas 4
   transições de status que já chamavam `auditService.log()` (Orçamento, Pedido de Venda, Requisição,
   Pedido de Compra), sempre só com o campo que mudou (`{ status: valor }` em cada lado, nunca um
   snapshot completo). Ordem de Produção e Revisão de BOM não tinham nenhuma chamada de auditoria no
   ponto de transição antes desta fase — não ganharam uma agora (adicionar uma chamada de auditoria nova
   onde nunca existiu seria além do escopo pedido: "os mesmos pontos que já estamos modificando"; essa
   lacuna específica permanece parte do débito catalogado na Fase 1, não foi criada por esta fase).
   Princípio 8 corrigido para registrar cumprimento parcial e onde exatamente termina.

4. **Separação Comercial/Industrial da Fase 4 original — resolvido em 2026-07-09.** A proposta original
   tinha 3 partes distintas; cada uma teve um destino diferente, verificado contra a implementação real:
   - **Snapshot de BOM na criação da OP**: **absorvido pela Fase 5** (ADR-006) —
     `ProductionOrder.bomRevisionId`, capturado e congelado na criação, cumpre exatamente o que a Fase 4
     original pedia, só que sob o nome da Reserva de Material em vez de "Comercial/Industrial".
   - **Ciclo de vida da OP em 6 estados** (Planejada→Liberada→Em Produção→Pausada→Finalizada→Encerrada):
     **substituído pelo modelo de 5 estados já implementado e validado na Fase 2** (ADR-002:
     `planned`/`in_progress`/`paused`/`completed`/`cancelled`). Esse modelo já está em produção, já foi
     aprovado pelo usuário na época sem ressalva, e nenhuma fase desde então sentiu falta dos estados
     "Liberada"/"Encerrada". **Decisão oficial**: a proposta de 6 estados está superada pela modelagem
     real da Fase 2 — encerrada, não será revisitada.
   - **Flag `AUTO_GENERATE_PRODUCTION_ORDER`/"Liberação da Produção" — encerrado definitivamente em
     2026-07-09.** Decisão oficial do usuário: esta proposta foi substituída pela arquitetura atual, não
     recuperada. A geração automática da Ordem de Produção na aprovação do Orçamento/Pedido de Venda
     (Fase 3, evento `orcamento.aprovado`) já é o fluxo natural e definitivo do Cozisteel ERP — não será
     introduzida uma etapa intermediária de "Liberação da Produção" só para manter compatibilidade com uma
     ideia antiga do roadmap. Removido da lista de exemplos do Princípio 9 (ver acima). Se no futuro
     surgir uma necessidade real de aprovação manual da produção, ela será tratada como uma **funcionalidade
     nova**, avaliada por seus próprios méritos — não como recuperação deste requisito.

5. **Campos de preparação Financeira** (`costCenterId`, `originDocumentType`/`originDocumentId`,
   `projectId`, `financialReferenceId`) — **resolvido em 2026-07-09**, ver
   [ADR-008](./ADR-008-infraestrutura-financeira.md): `originDocumentType`/`originDocumentId` cancelado
   definitivamente; `costCenterId`/`financialReferenceId` infraestrutura oportunista (mesmos gatilhos do
   item 1); `projectId` adiado sem janela prevista.

**Não são gaps** (verificado, decisão já fechada, citado aqui só para não serem confundidos com pendência):
geração automática de OP sem bloqueio por falta de material (Fase 5, validado explicitamente via
`AskUserQuestion` — comportamento definitivo, não um toggle de `SystemSetting` pendente); numeração de
documentos (`NumberingService`, já existe desde antes desta iniciativa).

## Revisão Arquitetural Final — Fases 1 a 5 (2026-07-09, antes da Fase 6)

Todos os 5 achados desta auditoria têm decisão registrada. Confirmação explícita pedida pelo usuário antes
de validar o ADR-007 e iniciar o MRP:

- **Os ADRs refletem fielmente a implementação.** Princípios 7, 8 e 9 deste documento foram corrigidos
  para bater com o código real (StatusHistory implementado, auditoria antes/depois parcial e explicitada,
  exemplo de `AUTO_GENERATE_PRODUCTION_ORDER` removido). ADR-002 recebeu uma entrada de log cruzando para
  a Fase 2.1. ADR-005/ADR-006/ADR-007/ADR-008 já estavam corretos e não precisaram de correção nesta
  rodada — conferido linha a linha contra o schema e os Services reais, não contra a memória do
  planejamento.
